# 监管端代码结构与审批接口方案

## 1. 现有代码结构分析

### 1.1 仓库形态

- 当前仓库是单模块 Android 工程，只有 `:app` 一个可运行模块。
- 技术栈是 `Activity + Fragment + ViewBinding + RecyclerView + DataStore`。
- 目前没有真正接入 Retrofit、Room、ViewModel、Repository 分层，业务数据以 `mock` 为主。

### 1.2 启动链路

真实启动路径如下：

1. `app/src/main/AndroidManifest.xml`
   - 启动页是 `com.qkzc.workerm.ui.auth.LaunchActivity`
2. `app/src/main/java/com/qkzc/workerm/ui/auth/LaunchActivity.kt`
   - 读取 `SessionStore`
   - 已登录跳 `MainActivity`
   - 未登录跳 `LoginActivity`
3. `app/src/main/java/com/qkzc/workerm/MainActivity.kt`
   - 承载底部导航
   - 切换 `HomeFragment / SupervisionFragment / TodoFragment / MessageFragment / ProfileFragment`

### 1.3 当前层次划分

- `data/session`
  - 登录态缓存和模拟登录逻辑
- `data/mock`
  - 首页、消息、监管等页面的假数据
- `data/home`
  - 首页 Dashboard 的 DTO 预埋，当前未真正接 API
- `ui/...`
  - 各个页面 Fragment
- `ui/common/adapter`
  - 通用 RecyclerView Adapter

### 1.4 当前结构优点

- 页面壳子已经齐全，做一阶段监管端原型够快。
- 首页、监管、消息、我的都已能直接扩展。
- `SessionStore` 已经把登录态和项目上下文持久化了，后续可以直接挂真实接口。

### 1.5 当前结构风险

- 现在业务数据直接从 `mock` 进页面，后面接接口时会出现 UI 和数据层耦合。
- `TodoFragment` 之前只是静态待办，没有审批模型、审批状态、审批动作和留痕概念。
- 仓库里虽然预埋了 `data/home/model`，但还没有统一返回体、接口服务、仓库层和错误处理。

### 1.6 本次改造的安全落点

- 首页快捷入口：继续放在 `HomeFragment + SupervisionMockData`
- 审批中心页面：直接落在 `TodoFragment`
- 审批数据模型：新增 `data/approval`
- 审批列表渲染：新增 `ui/common/adapter/ApprovalAdapter`

这条路径改动最小，不会破坏现有底部导航结构。

## 3. 审批业务边界

### 3.1 业务角色

- 工人端：发起申请
- 监管端 App：审批申请
- 后台管理：查看审批记录、配置审批流程、补查异常单据
- 后端：负责流程流转、权限校验、审批留痕、消息通知

### 3.2 一期建议先支持的申请类型

- 补卡申请
- 入场申请
- 请假申请
- 退场申请

不要一开始把所有业务都做成通用表单引擎。先把这 4 类跑通，后面再扩。

### 3.3 状态流转

建议统一状态：

- `DRAFT`：草稿
- `PENDING`：待审批
- `APPROVED`：审批通过
- `REJECTED`：驳回
- `CANCELLED`：申请人撤销
- `ARCHIVED`：归档

监管端 App 主要关注：

- `PENDING`
- `APPROVED`
- `REJECTED`

## 4. 后端表结构设计

### 4.1 业务申请主表 `biz_apply_order`

适合放通用主信息。

```sql
CREATE TABLE biz_apply_order (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '申请单ID',
    apply_no VARCHAR(32) NOT NULL COMMENT '申请单号',
    apply_type VARCHAR(32) NOT NULL COMMENT '申请类型：FILL_CARD/ENTRY/LEAVE/EXIT',
    applicant_user_id BIGINT NOT NULL COMMENT '申请人用户ID',
    applicant_name VARCHAR(64) NOT NULL COMMENT '申请人姓名',
    project_id BIGINT NOT NULL COMMENT '项目ID',
    project_name VARCHAR(128) NOT NULL COMMENT '项目名称',
    team_id BIGINT DEFAULT NULL COMMENT '班组ID',
    team_name VARCHAR(128) DEFAULT NULL COMMENT '班组名称',
    biz_id BIGINT NOT NULL COMMENT '业务表ID',
    status VARCHAR(16) NOT NULL COMMENT '状态',
    current_node_no INT DEFAULT 1 COMMENT '当前审批节点',
    current_approver_id BIGINT DEFAULT NULL COMMENT '当前审批人ID',
    current_approver_name VARCHAR(64) DEFAULT NULL COMMENT '当前审批人',
    apply_reason VARCHAR(500) DEFAULT NULL COMMENT '申请说明',
    submit_time DATETIME DEFAULT NULL COMMENT '提交时间',
    finish_time DATETIME DEFAULT NULL COMMENT '处理完成时间',
    version INT NOT NULL DEFAULT 0 COMMENT '乐观锁版本号',
    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_apply_no (apply_no),
    KEY idx_project_status (project_id, status),
    KEY idx_applicant_status (applicant_user_id, status),
    KEY idx_approver_status (current_approver_id, status)
) COMMENT='审批申请主表';
```

### 4.2 审批节点表 `biz_apply_approval_node`

记录每个节点。

```sql
CREATE TABLE biz_apply_approval_node (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '节点ID',
    apply_order_id BIGINT NOT NULL COMMENT '申请单ID',
    node_no INT NOT NULL COMMENT '节点顺序',
    node_name VARCHAR(64) NOT NULL COMMENT '节点名称',
    approver_user_id BIGINT NOT NULL COMMENT '审批人ID',
    approver_name VARCHAR(64) NOT NULL COMMENT '审批人姓名',
    action VARCHAR(16) DEFAULT NULL COMMENT '动作：APPROVE/REJECT/TRANSFER',
    action_remark VARCHAR(500) DEFAULT NULL COMMENT '审批意见',
    action_time DATETIME DEFAULT NULL COMMENT '审批时间',
    status VARCHAR(16) NOT NULL COMMENT 'PENDING/FINISHED/SKIPPED',
    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    KEY idx_apply_node (apply_order_id, node_no),
    KEY idx_approver_pending (approver_user_id, status)
) COMMENT='审批节点表';
```

### 4.3 审批动作日志表 `biz_apply_action_log`

只做留痕，不参与当前节点判断。

```sql
CREATE TABLE biz_apply_action_log (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '日志ID',
    apply_order_id BIGINT NOT NULL COMMENT '申请单ID',
    operator_user_id BIGINT NOT NULL COMMENT '操作人ID',
    operator_name VARCHAR(64) NOT NULL COMMENT '操作人姓名',
    action VARCHAR(16) NOT NULL COMMENT 'SUBMIT/APPROVE/REJECT/CANCEL/TRANSFER',
    action_remark VARCHAR(500) DEFAULT NULL COMMENT '说明',
    from_status VARCHAR(16) DEFAULT NULL COMMENT '原状态',
    to_status VARCHAR(16) DEFAULT NULL COMMENT '新状态',
    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    KEY idx_apply_time (apply_order_id, create_time)
) COMMENT='审批动作日志表';
```

### 4.4 业务明细表

不同申请类型建议各自建表，不建议把所有字段塞进一个 JSON。

例如：

- `attendance_fill_card_apply`
- `worker_entry_apply`
- `worker_leave_apply`
- `worker_exit_apply`

原因很直接：

- 字段差异大
- 查询条件不同
- 后台统计不同
- 后续校验规则不同

## 5. 接口设计

建议接口按端区分：

- 工人端：`/app/worker/...`
- 监管端：`/app/supervision/...`
- 后台管理：`/admin/...`

### 5.1 工人端提交申请

#### 1. 创建申请

`POST /app/worker/apply-orders`

请求示例：

```json
{
  "applyType": "FILL_CARD",
  "projectId": 1001,
  "bizData": {
    "attendanceDate": "2026-04-20",
    "fillCardType": "OFF_DUTY",
    "fillTime": "2026-04-20 21:30:00",
    "reason": "昨晚手机损坏，未能完成下班打卡"
  }
}
```

返回示例：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "applyOrderId": 90001,
    "applyNo": "BK20260421001",
    "status": "PENDING"
  }
}
```

#### 2. 我的申请列表

`GET /app/worker/apply-orders/my?pageNum=1&pageSize=20&status=PENDING`

#### 3. 我的申请详情

`GET /app/worker/apply-orders/{applyOrderId}`

#### 4. 撤销申请

`POST /app/worker/apply-orders/{applyOrderId}/cancel`

适用前提：

- 当前状态必须还是 `PENDING`
- 只能申请人本人撤销

### 5.2 监管端审批接口

#### 1. 待我审批列表

`GET /app/supervision/approvals/pending?pageNum=1&pageSize=20&applyType=FILL_CARD&projectId=1001`

返回字段建议：

```json
{
  "list": [
    {
      "applyOrderId": 90001,
      "applyNo": "BK20260421001",
      "applyType": "FILL_CARD",
      "applyTypeName": "补卡申请",
      "title": "王磊 4 月 20 日晚班补卡",
      "applicantUserId": 3001,
      "applicantName": "王磊",
      "projectId": 1001,
      "projectName": "一号工地",
      "currentNodeName": "项目经理审批",
      "applyReason": "昨晚手机损坏，未能完成下班打卡",
      "submitTime": "2026-04-21 08:20:00"
    }
  ],
  "total": 1
}
```

#### 2. 我已审批列表

`GET /app/supervision/approvals/processed?pageNum=1&pageSize=20&result=APPROVED`

#### 3. 审批详情

`GET /app/supervision/approvals/{applyOrderId}`

返回要包含三块：

- 申请主信息
- 业务明细
- 审批轨迹

#### 4. 审批通过

`POST /app/supervision/approvals/{applyOrderId}/approve`

请求示例：

```json
{
  "remark": "同意，按流程执行",
  "version": 3
}
```

#### 5. 审批驳回

`POST /app/supervision/approvals/{applyOrderId}/reject`

请求示例：

```json
{
  "remark": "缺少实名制照片，请补充后重提",
  "version": 3
}
```

#### 6. 审批统计

`GET /app/supervision/approvals/statistics?projectId=1001`

返回建议：

```json
{
  "pendingCount": 12,
  "processedTodayCount": 8,
  "rejectedCount": 2
}
```

### 5.3 后台管理接口

后台主要不是做动作，而是查记录和补救。

建议提供：

- `GET /admin/approvals/page`
- `GET /admin/approvals/{applyOrderId}`
- `GET /admin/approvals/{applyOrderId}/logs`
- `POST /admin/approvals/{applyOrderId}/transfer`
- `POST /admin/approvals/{applyOrderId}/force-close`

## 6. 后端处理流程

### 6.1 工人端提交申请

1. 校验申请类型和业务参数
2. 写业务明细表
3. 生成 `biz_apply_order`
4. 生成审批节点
5. 记录 `SUBMIT` 日志
6. 推送监管端待审批消息

### 6.2 监管端审批通过

1. 校验当前登录人是否是当前节点审批人
2. 校验申请单状态必须是 `PENDING`
3. 用 `version` 做乐观锁更新
4. 更新当前节点为 `FINISHED`
5. 若还有下一个节点：
   - 推进到下一节点
   - 更新主表 `current_node_no / current_approver_id`
6. 若已经最后一节点：
   - 主表改为 `APPROVED`
   - 写 `finish_time`
   - 回写对应业务表状态
7. 写 `APPROVE` 日志
8. 通知申请人

### 6.3 监管端驳回

1. 校验权限和状态
2. 当前节点写驳回意见
3. 主表改为 `REJECTED`
4. 回写业务表为驳回
5. 写 `REJECT` 日志
6. 通知申请人补资料或重新提交

## 7. 关键校验点

### 7.1 权限

- 监管端审批接口必须按 `当前项目 + 当前角色 + 数据范围` 校验
- 不能只靠前端隐藏按钮

### 7.2 并发

- 审批动作必须带 `version`
- 避免两个人同时处理同一条审批单

### 7.3 留痕

至少要留 3 类记录：

- 谁提交的
- 谁审批的
- 每次状态怎么变的

### 7.4 业务回写

审批单通过后，不能只改审批表。

必须同步更新对应业务表，例如：

- 补卡申请通过后，写入考勤补卡结果
- 入场申请通过后，开通工人门禁/项目状态
- 请假申请通过后，更新请假单状态

## 8. 当前阶段最推荐方案

### 方案 A：最推荐，适合当前阶段

- 申请主表 + 节点表 + 动作日志表
- 业务明细分表
- 工人端提交
- 监管端审批
- 后台只做查询和补救

优点：

- 足够稳
- 后面容易扩类型
- 数据也清晰

### 方案 B：后期扩展

- 再加审批模板配置表
- 支持项目级流程自定义
- 支持抄送、转交、多节点会签

### 方案 C：不推荐但可备选

- 所有申请都塞到一个大 JSON 表里

问题：

- 校验难
- 查询慢
- 后续一定返工

## 9. 对当前仓库的下一步建议

接下来建议按这个顺序继续落：

1. 给 `ApprovalRepository` 补真实 Retrofit 接口和 DTO。
2. 新增 `ApprovalApi / ApprovalRepositoryImpl / ApprovalMapper`。
3. 把 `TodoFragment` 改成 `ViewModel + State`，不要再直接读内存对象。
4. 后端先把补卡申请和入场申请两条审批链打通。
5. 后台管理只要先补分页查询、详情、日志查看即可，不要先做复杂流程配置。
